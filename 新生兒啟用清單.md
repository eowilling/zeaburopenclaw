# OpenClaw 新生兒啟用清單

這份專案剛從模板建立，像新生兒一樣很多功能還不能直接用。底下是**要讓它真正能跑起來**需要完成的項目，依優先順序排列。

---

## 一、目前已有、可直接用的部分

| 項目 | 說明 |
|------|------|
| **openclaw.json** | 已含 Telegram、LINE bot token、Gateway 設定、Control UI 免裝置配對、Gemini 型號與 workspace 路徑 |
| **workspace-seed** | 內含 MEMORY.md（elior 稱呼規則、InsForge、LINE 429 等） |
| **Dockerfile + entrypoint** | 部署時會把設定與 workspace 種到磁碟 |
| **文件** | Zeabur 設定、GitHub 部署、ENV、Admin API 用量、InsForge 稱呼等說明 |

---

## 二、還沒辦法用／需要你先設定的部分

### 1. Gemini API Key（必做，否則無法回話）

- **狀況**：`openclaw.json` 裡指定用 `google/gemini-2.5-flash`，但 **API 金鑰**要由環境變數提供。
- **做法**：在 Zeabur 服務的 **Variables** 新增至少一組：
  - `GEMINI_API_KEY_1`（或 `GOOGLE_GENERATIVE_AI_API_KEY`，依你 OpenClaw 版本支援的變數名）
  - 若要輪替多組 key，可設 `GEMINI_API_KEY_1`～`GEMINI_API_KEY_5`，詳見 **ENV_五組Gemini_API_Key.md**
- **取得方式**：用 Google 帳號到 [Google AI Studio](https://aistudio.google.com/apikey) 申請 API Key 後貼上。

---

### 2. Zeabur 部署必做設定

| 項目 | 說明 | 對應文件 |
|------|------|----------|
| **硬碟掛載** | 必須把 Volume 掛到 **`/home/node`**，設定與 workspace 才會持久化，重啟才不會消失 | Zeabur設定說明.md § 一 |
| **OPENCLAW_CONFIG_PATH** | 環境變數設為 **`/home/node/.openclaw/openclaw.json`**，Gateway 才會讀磁碟上的設定（含免裝置配對） | Zeabur設定說明.md § 三、§ 八 |
| **OPENCLAW_GATEWAY_TOKEN** | 與 `openclaw.json` 裡 `gateway.auth.token` 相同，Control UI 與程式連線用 | Zeabur設定說明.md § 三 |
| **啟動指令** | 使用文件中的 startup + gateway 指令，讓 entrypoint 正確複製種子並啟動 | Zeabur設定說明.md § 二 |

沒做上述幾項，常見問題是：**Control UI 連線出現 `device identity required`（1008）**，或重啟後設定被清掉。

---

### 3. Control UI 能連、能改設定

- **狀況**：要能用瀏覽器開 Zeabur 給的網址、輸入 token 連上並改設定。
- **檢查**：
  - 環境變數已設 `OPENCLAW_CONFIG_PATH`、`OPENCLAW_GATEWAY_TOKEN`
  - 在「設定」頁改一項後儲存，重啟服務再開，確認變更仍在
詳見 **Zeabur設定說明.md** 的檢查清單與故障排除。

---

### 4. 瀏覽器 / 沙箱（可選）

- **狀況**：`openclaw.json` 裡 `browser` 設為 `attachOnly`，並指向 `http://openclaw-sandbox-browser:9222`。
- **意義**：若沒有在 Zeabur 同專案部署 **openclaw-sandbox-browser** 這類服務，瀏覽器相關功能會連不到，屬正常；不影響純文字對話。
- **若要使用**：需在 Zeabur 加建對應的 browser 服務，並確保網路可連到該 URL。

---

### 5. 用戶稱呼（InsForge，可選）

- **狀況**：MEMORY.md 寫了「用 InsForge identities 的 nickname 稱呼用戶」，但 elior 要能**連到 InsForge** 才會真的去查。
- **做法**：
  - 若有 **insforge-identities** 技能：在設定裡啟用，並在 Zeabur 設 `INSFORGE_BASE_URL`、`INSFORGE_ANON_KEY`。
  - 若沒有該技能：可用 web-fetch 或自建小 API 查 identities 表，並在 MEMORY 裡寫清楚流程。
詳見 **InsForge-用戶稱呼設定.md**。

---

### 6. Telegram / LINE 的 token 安全（建議）

- **狀況**：目前 Telegram bot token、LINE channel token/secret 是直接寫在 `openclaw.json` 裡，若 repo 公開會有風險。
- **建議**：
  - 改為用環境變數注入（若 OpenClaw 支援從 env 讀 channel 設定）。
  - 或至少不要把含真實 token 的 `openclaw.json` push 到公開 GitHub，可改用「從 Zeabur 抓回來的設定」只放本機或私密 repo。

---

## 三、建議完成順序

1. **Zeabur**：掛 Volume `/home/node`、設好 `OPENCLAW_CONFIG_PATH`、`OPENCLAW_GATEWAY_TOKEN`、啟動指令。
2. **Gemini**：在 Variables 設好至少一組 `GEMINI_API_KEY_1`（或你版本對應的變數名）。
3. **驗證**：重新部署後，用 Control UI 連線並在設定頁改一項儲存，重啟後確認仍在。
4. **對話**：在 Telegram 或 LINE 對 bot 發訊息，確認有正常回覆。
5. （可選）瀏覽器服務、InsForge 稱呼、token 改由環境變數等，再依需求補上。

## 五、Log 常見訊息說明

### delivery-recovery: exceeded max retries (5/5) — moving to failed/

- **意思**：之前有幾則「要回給用戶的訊息」（delivery）送不出去（例如當時 API 429、網路斷、bot 重啟），已經重試 5 次仍失敗，啟動時被標成 failed 並跳過。
- **要不要處理**：**不用特別處理**。那是**舊的**待送件，清掉就沒了；之後只要 API 與連線正常，新的回覆會正常送出，不會一直累積。
- **entrypoint**：本專案 `entrypoint.sh` 每次啟動會清空 `delivery-queue`，所以新的 pending 不會留到下次；若 log 仍出現「moving to failed」，代表是上次或更早之前就超過 5 次重試、被持久化下來的紀錄，Gateway 只是在報告「這幾筆我放棄了」。

完成上述必做項目後，你的 OpenClaw 就不再是「新生兒」、可以正常對話與用 Control UI 管理設定了。
